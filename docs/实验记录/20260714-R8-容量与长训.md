# R8：容量不足 + 早停过早 — 诊断与加深长训

## 用户指出的问题（正确）

1. **显存只占 ~1GB / 24GB**：不是“省着用”，是模型太浅 + encoder 曾有 Python 逐样本循环，GPU 根本吃不饱。
2. **早停常在 ep15–30**：`patience=6–8` + `max_epochs=16–24` 会在粗精度平台就停，来不及细收敛。

## 定量诊断

| 项 | 旧冠军 / R7 | 含义 |
|---|---|---|
| 参数量 | **≈2.3M** | 对 100k 回归过浅 |
| encoder | 浅 3 段 MLP + **Python for 循环 featurize** | 主机同步 → util 常 <20% |
| head | Linear→256→6（2 层） | 容量集中在路由头，表达力弱 |
| best epoch | 13–20 | 曲线还在缓降就被 patience 砍掉 |
| 峰值显存 | ~0.5–1.4GB | bag-MLP 激活极小，VRAM≈参数+Adam |

说明：histogram 输入是固定 ~281 维 bag，**即使用大 batch，激活内存也上不去**；要吃显存必须加宽/加深参数量。探针：~235M ≈4.5GB；~700M ≈13GB。

## 已落地修改

1. **向量化** `InverseD2HistogramEncoder._featurize_batch`（去掉训练路径上的 for/`.item()`）。
2. **加深 encoder**：`histogram_num_blocks` residual MLP（默认 0=旧浅网兼容）。
3. **加深 head**：`head_num_layers`（默认 2=兼容）。
4. **新 100k 配方** `configs/scale_100k_r8_deep_long.yaml`：
   - ~235M（hidden_enc=2048×12 blocks，head=1024×4）
   - batch=2048，lr=0.005（sqrt 缩放）
   - max_epochs=**100**，patience=**25**，warmup=5
   - dropout=0.15，weight_decay=0.05（防大模型过拟合）

## Gate

vs R7 baseline valid1400：elem **15.36%**，angle MAE **8.68°**，length MAE **0.866Å**。

- 通过：elem ≥ **17%** 或 angle MAE ≤ **7.5°**（任一明显进步即可继续加宽到 ~700M / 全量）。
- 失败：大模型不涨或明显过拟合 → 缩回容量；按 v2 方案进入规范胞 + 输出头减法，而不是继续堆宽或跳 Phase B 多假设。
- 映射：本实验对应方案 **R11b / 主线①**（见 `20260714-CellIndexing-Top1逐步优化方案v2.md`）。
