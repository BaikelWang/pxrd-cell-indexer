# R7：训练策略 bug 修复 + peak_consistency 公平复核

## 背景

R6-A 的 100k 峰一致性实验把 `batch_size` 从冠军的 64 直接提到 1024（为了「用满 24GB」），但学习率仍是
`0.001` 不变。结果 epoch5 的 elem 只有 0.7%（冠军同期 11%），几乎不可用——不是 loss 坏了，是大 batch
后每个 epoch 的梯度更新次数少了 16 倍，等效学习率不够，训练明显欠拟合。

同时发现 `peak_consistency_loss`（`losses.py`）内部是纯 Python `for i in range(batch)` 循环，每个样本
单独做 3×3 矩阵求逆 + `.item()` 同步，batch 越大这个循环越慢、GPU 利用率越低（实测只有 16-20%）。

## 本轮做的事

1. **向量化 `peak_consistency_loss`**：去掉 python for 循环，晶格矩阵构建 (`_batch_lattice_params_to_matrix`)、
   `G*` 求逆、hkl 理论 1/d² 计算全部批量化，只保留一次 padding + mask，无 host 同步。
   - 单测 4 个全过（数值与旧实现一致）。
   - 微benchmark：batch=1024 时 fwd+bwd 约 3.4ms/iter（297 iter/s），此前的循环实现是主要瓶颈来源。
2. **LR-scaling 探针**（100k，5 epoch，`mode=baseline`，与冠军的 bs64/lr1e-3 逐 epoch 对比）：
   - `bs256 / lr=0.001`（不变）：ep5 elem=9.07%，略慢于冠军（11.00%），符合「大 batch 更新次数少」预期，但没有崩。
   - `bs256 / lr=0.002`（sqrt 缩放，4× batch → ×2 lr）：ep4 elem=11.21%、ep5 elem=11.43%，**追平甚至略超冠军**（9.36%/11.00%）。
   - 结论：sqrt-scaling 是可行的 batch/LR 配比修复方案，采用 `batch=256, lr=0.002, num_workers=16`。
3. **公平复核（同一套 batch/LR 配方，24 epoch，patience 8，并行跑在同一张卡上，GPU 利用率 ~94%）**：
   - `scale_100k_r7_baseline_bs256`（`mode=baseline`，对照组）
   - `scale_100k_r7_peak_w005_bs256`（`mode=peak_consistency`，λ=0.05）

## 结果

| 配置 | best epoch | elem | mapping | angle MAE | length MAE | length MAPE | loss |
|---|---|---|---|---|---|---|---|
| 旧冠军 bs64/lr1e-3（16 epoch） | 13 | 15.43% | 15.93% | 8.71° | 0.897Å | 14.0% | 0.299 |
| 新 baseline bs256/lr2e-3（24 epoch） | 20 | **15.36%** | 16.14% | 8.68° | 0.866Å | 13.7% | 0.290 |
| peak_w005 bs256/lr2e-3（24 epoch） | 19 | 14.43% | 15.43% | 8.90° | 0.877Å | 13.9% | 0.298 |

## 结论

1. **训练策略 bug 已修复并验证**：`batch=256 + lr=0.002`（sqrt 缩放）+ `num_workers=16` 的新 baseline，
   在完全独立训练（24 epoch, seed42）后精度与旧冠军（15.36% vs 15.43%）几乎相同——在噪声范围内一致，
   证明大 batch 本身没有伤精度，纯粹是之前没配对 LR 的锅。副作用是单 epoch 优化步数少了 4 倍、
   两个实验并行跑仍能把 GPU 利用率打到 94%（此前只有 16-20%），总训练墙钟时间也明显缩短。
2. **peak_consistency（λ=0.05）在公平对比下没有带来精度提升**：angle MAE / length MAE / length MAPE
   三个物理误差指标都和新 baseline基本一致甚至略差（8.90° vs 8.68°，0.877Å vs 0.866Å），
   elem 也更低（14.43% vs 15.36%）。这和 R6-A P0-700 过拟合阶段「可学、稳定」的结论不矛盾——
   小样本上loss 能降，但在真实 10 万级数据上，它没有在原有 SmoothL1 之上带来额外的泛化增益，
   反而因为对每个样本单独算 obs↔theory 的软 Chamfer，多引入了一些梯度噪声。
   **裁决：peak_consistency λ=0.05 不采纳**，不满足「大幅进步才上全量数据」的门槛。
3. 新 baseline（bs256/lr2e-3）本身可以作为往后所有实验的默认训练配方替代旧的 bs64/lr1e-3——
   同精度、同或更快墙钟时间、GPU 利用率更高，后续所有 100k/全量实验建议统一切到这套配方。

## 下一步建议

- 不要用 peak_consistency 去跑全量（600万条）数据，性价比不够。
- 本记录对应 v2 方案 **主线③**；后续按 Phase A：头减法（R10）→ Encoder 加深（R11b，含进行中的 deep_long）→ Phase A 出口后再谈全量 / Phase B。
- 新 baseline（bs256/lr2e-3）保留为默认训练配方。
