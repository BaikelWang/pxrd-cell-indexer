## 1. 排查清单（按优先级）

### T1. 几何误差分解（已做 ✅）

**问题**：严口径失败，主要是角度不够还是边长不够？  
**方法**：对 MP100 stored `raw_pred` 统计 max 角度误差 / max 相对边长误差 vs 0.05/3°。  
**判定**：若绝大多数样本角或边远超阈值 → 确认「回归精度」是根因，而非 FOM。

**结果（baseline）**：

| 指标 | 数值 |
|---|---:|
| 严口径 raw / fom | 3% / 13% |
| raw max-angle median / p90 | **18.4° / 30.7°** |
| raw max-len-rel median / p90 | **31.8% / 51.0%** |
| 边长已进 5%（max axis） | **仅 5%** |
| 角度已进 3°（max angle） | **仅 16%** |
| 角+边同时进阈值（逐参数） | **仅 2%** |

**结论**：**角和边都远不够**；边长相对误差中位数已 ~32%，比 5% 目标差一个数量级。`length_angle` 角度略好（进 3° 的比例 16%→26%），但边长更差（进 5% 的比例 5%→0%），与严口径 raw 0% 一致。

---

### T2. Bravais 搜索半径是否太小（已做 ✅）

**问题**：在 raw 附近做 8 假设 snap，严口径下能否捞到真解？  
**方法**：对每个 raw，检查 best-snap / 任一 snap 是否 `find_mapping` @ 0.05/3°。

**结果（baseline）**：

| 检查 | 命中率 |
|---|---:|
| best-snap 严口径 | **3%**（≈ raw，几乎无增益） |
| **任一 8 snap 严口径（oracle）** | **仅 12%** |
| best 为 identity | 25% |

**结论**：即使穷举当前 Bravais 集合，严口径召回也只有 **12%** ≪ 引擎 66%。**搜索空间过窄已证实**；不是「FOM 没选对 snap」。

---

### T3. Top-K 池在严口径下的真实召回（已做 ✅）

**问题**：宽松 topk~100% 是否掩盖了严口径池质量？  
**方法**：用 stored raw 重建 `build_top_k_candidates(K=20)`，算 topk @ 两档容差。

**结果（baseline）**：

| 口径 | topk |
|---|---:|
| 0.3 / 10° | **100%** |
| **0.05 / 3°** | **26%** |

**结论**：候选生成在严口径下只覆盖约 1/4 样本 → **FOM 再强也最多 ~26% 上界（若排序完美）**；实际 fom 13% 说明排序还有一半空间，但主矛盾仍是池子。

---

### T4. FOM 公式是不是主瓶颈（已做 ✅）

**问题**：换 `strict_dewolff` / `intensity_weighted` 能否在严口径下明显涨点？  
**方法**：同一候选池，三种 FOM mode 重排。

**结果（baseline，严口径 Top-1）**：

| FOM mode | Top-1 |
|---|---:|
| heuristic（默认） | **13%** |
| strict_dewolff | **13%** |
| intensity_weighted | **13%** |
| **把真解注入池后再 heuristic** | **100%** |

**结论**：
1. 换 FOM 公式 **零增益** → 当前严口径下 **FOM 不是主瓶颈**。  
2. 真解在池内时 FOM 能 100% 选中 → FOM 在 ideal 峰上「会认人」；缺的是**把真解放进池子**（回归+搜索）。

---

### T5. 2θ 取整信息损失（理论估算已做 ✅；训练消融待 Phase3 后）

**问题**：Encoder `pxrd_x.long()` 是否从信息上就不可能达到 3°/5%？  
**方法**：估算 0.5° 截断引起的 |ΔQ|，对比 FOM `q_tol=1e-6`。

**结果**：2θ=10–80° 时，最坏 |ΔQ| ≈ **0.014–0.018 Å⁻¹**，比默认 Q 匹配容差大 **~10⁴ 倍**。

**结论**：取整对峰匹配是**巨大**信息损失；但是否为当前 13% 的主因，需 Phase3 后做「连续 2θ 输入」消融才能定量。现阶段记为 **高优先级嫌疑**。

---

### T6. 晶系难度（已做 ✅，强化）

**结果（baseline 严口径 fom，粗标签）**：trigonal/rhom 38 个里 fom 21%；hex 16 个仅 6%；cubic/mono 样本少且 0%。  
结合 valid 宽松 proxy：hex/trig 3–7% vs cubic 76%。

**结论**：低对称失败是稳定现象；后续数据/算法必须单独盯 hex/trig/mono/tric。

## 2. 已确认的问题（证据链）

```text
严口径失败
  ├─ 回归几何太粗（T1）：角 median 18°、边 median 32%  ≫ 3°/5%
  ├─ Bravais 搜不到（T2）：8 snap oracle 仅 12%
  ├─ Top-K 池不够（T3）：strict topk 26%（宽松 100%）
  ├─ FOM 不是主因（T4）：三 mode 全是 13%；真解在池则 100%
  └─ 2θ 取整可疑（T5）：ΔQ ~1e-2 vs q_tol 1e-6（待训练消融定量）
```

**对「完胜 JADE/Mc」的含义**：

1. **先把严口径 topk 从 26% 拉到 70%+**（回归更准 和/或 搜索更大），否则理论天花板低于引擎。  
2. **FOM 微调可以往后放**——池子修好前收益≈0。  
3. **Phase2 loss 消融未解决根因**——`length_angle` 严口径更差，与 T1 边长恶化一致。

## 5. 第二轮排查结果（2026-07-09，Phase3 仍在跑）

> GPU 仍有余量时完成：valid 严口径、尺度变体、容差曲线、峰–标签错位、FOM 在密池上的行为、**find_mapping 子胞伪命中**。

### T7a. valid1400 @ 0.05/3°（baseline ckpt）✅

| 指标 | 0.3/10° | **0.05/3°** |
|---|---:|---:|
| raw_top1 | 40.9% | **6.2%** |
| fom_top1 | 85.9% | **8.6%** |
| topk | 99.1% | **16.5%** |

产物：`results/diagnostics_phase3_wait/valid1400_baseline_ltol0.05_atol3.json`  
**结论**：不是 MP100 独有崩盘——**in-distribution valid 在严口径下同样塌到个位数**。训练选模用的宽松尺子与真实目标严重脱节。

### T7b. 尺度变体 vs 仅 snap（MP100，stored raw）✅

| 候选池 | 宽松 topk | 严口径 find_mapping topk |
|---|---:|---:|
| 仅 8 Bravais snap | 81% | 12% |
| Top-20 无尺度变体 | 93% | 16% |
| Top-20 默认尺度 | 100% | 26% |
| Top-40 + 更密尺度（含 ×3/4/1/3/1/4…） | 100% | **95%** ← 看似神了 |

密池 FOM Top-1（严口径）：**30%**（默认池 13%）。  
被「救回」的样本几乎全是 `scale=0.25 / 0.33` 的**缩小胞**。

### T7b+ 关键新问题：`find_mapping` 子胞伪命中 ✅

对密池 95 个「严口径 topk 命中」再查**逐参数**是否在 0.05/3° 内：

| 检查 | 数量 |
|---|---:|
| `find_mapping` 命中 | 95 |
| 其中 a,b,c,α,β,γ **逐参数**都进 0.05/3° | **仅 4** |
| 仅靠 mapping（子胞/倍胞等价） | **91** |
| mapping-only 的 vol(cand)/vol(truth) 中位数 | **0.021**（约 1/48 体积） |

默认 K=20：mapping 命中 26，逐参数命中仅 **7**。

**结论（极重要）**：
1. 加密尺度把 topk「拉到 95%」**大部分是评测口径把子胞算作命中**，不是回归突然变准。  
2. FOM 在密池上只到 30%，且失败时 **97%** 是错误候选 `n_matched` 更高；赢家多为 `scale=3/4` 大胞，输家多为 `scale=0.25` 子胞——FOM 其实在**拒绝**那些 mapping 伪命中。  
3. 因此：**不能只报 find_mapping@0.05/3°**；至少并行报「逐参数命中」或限制体积比，否则扩搜索会被假阳性误导。

### T7c. 容差工作曲线（MP100 baseline）✅

| ltol / atol | raw | fom | topk |
|---|---:|---:|---:|
| **0.05 / 3°** | 3% | 13% | 26% |
| 0.05 / 5° | 7% | 22% | 35% |
| 0.05 / 10° | 8% | 48% | 66% |
| 0.10 / 5° | 11% | 54% | 76% |
| 0.15 / 5° | 19% | 68% | 84% |
| 0.20 / 8° | 34% | 82% | 98% |
| **0.30 / 10°** | 58% | 88% | 100% |

**结论**：模型「舒服区」大约在 **ltol≳0.2、atol≳8°**；引擎硬指标 0.05/3° 远在舒适区外。边长容差比角度容差更卡 raw（0.30/3° raw 仍仅 29%）。

### T7d. 峰–标签设定错位（MP100 CIF）✅

raw 相对各设定的误差（mean）：

| 标签设定 | len_mae | ang_mae | len_mape | 严口径 raw |
|---|---:|---:|---:|---:|
| primitive（=json truth） | **1.59** | 10.3° | **18.9%** | 3% |
| reduced（峰模拟路径） | 2.49 | **8.6°** | 24.7% | 3% |
| conventional | 2.76 | 9.4° | 29.0% | 3% |

raw 几何上更接近 reduced / conventional 的比例仅 **21% / 15%**。  
**结论**：D1「峰来自 reduced、标签 primitive」**不是当前严口径崩盘的主因**——模型整体仍更贴 primitive 标签，只是贴得不够紧。该项优先级下调。

### 额外：长度尺度偏置

MP100 raw 相对 truth 的边长几何平均比：median **0.89**（略偏小），~1× 占 34%；未见系统性整倍胞（2×/3×）主导。与「密池靠 ×0.25 子胞骗 mapping」是不同机制。

## 6. 本轮新确认的算法/评测问题（汇总）

| ID | 问题 | 证据强度 | 含义 |
|---|---|---|---|
| **N1** | 严口径在 valid 与 MP100 双崩 | 极强 | 全局几何精度问题，非 OOD 特例 |
| **N2** | `find_mapping` 把子胞/倍胞算命中，膨胀 topk | 极强 | 扩搜索前必须加「逐参数/体积比」护栏 |
| **N3** | 默认尺度集合偏窄；但盲目加密会被 N2 误导 | 强 | 要扩搜索，需与真 primitive 体积一致的约束 |
| **N4** | 池变大后 FOM 仍弱（30% vs 95% 伪池） | 强 | 大胞假峰匹配赢过真（子）胞；需更好 FOM 或体积先验 |
| **N5** | 峰–标签设定错位非主因 | 中 | 可暂缓 D1 重标定 |
| **N6** | 舒适容差 ≫ 引擎硬指标 | 强 | 必须改训练目标/选模尺子 |

### 对下一步的修正建议

1. **评测**：主报 `find_mapping@0.05/3°` 时，强制附带 `elementwise@0.05/3°` 或 `|log(V_pred/V_truth)|<log(1.5)` 过滤。  
2. **搜索**：可试加密尺度，但候选需体积/Niggli 约束，避免子胞刷分。  
3. **训练**：valid 严口径（及逐参数 MAE）进监控；别再只看 0.3/10°。  
4. **FOM**：在「真池」上再调；当前密池 30% 含大量伪命中，数字谨慎解读。
